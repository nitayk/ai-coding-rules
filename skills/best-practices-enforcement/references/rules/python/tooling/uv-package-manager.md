# uv: The Single Python Toolchain

`uv` is Astral's Rust-based package and project manager. It is "a single tool to replace pip, pip-tools, pipx, poetry, pyenv, twine, virtualenv, and more" and runs 10-100x faster than pip ([uv docs](https://docs.astral.sh/uv/)). For new projects and CI, prefer `uv` over any of the tools it replaces.

---

## When to use uv

| Task | Old tool(s) | uv command |
|---|---|---|
| Create venv | `python -m venv` / `virtualenv` | `uv venv` |
| Install deps | `pip install -r requirements.txt` | `uv sync` / `uv pip install` |
| Add a dep | `poetry add` / manual edit + reinstall | `uv add <pkg>` |
| Install CLI tool | `pipx install` | `uv tool install <pkg>` |
| Run script with deps | `pip install + python` | `uv run <script.py>` |
| Manage Python versions | `pyenv` | `uv python install 3.13` |
| Lockfile | `poetry.lock` / `pip-compile` | `uv.lock` (auto-managed) |
| Publish to PyPI | `twine upload` | `uv publish` |

## Project initialization

```bash
# Good: uv init creates pyproject.toml + .venv + uv.lock layout
uv init my-service
cd my-service
uv add fastapi 'pydantic>=2'
uv add --dev pytest ruff mypy

# Good: pin Python interpreter for the project
uv python pin 3.13

# Good: reproducible install from lockfile (CI + dev)
uv sync --frozen   # fail if uv.lock is out of sync with pyproject.toml
```

The `uv.lock` file is multi-platform and should be committed. Re-resolving (`uv lock`) is explicit; `uv sync` will not silently re-resolve unless `pyproject.toml` changed.

## Dependency groups (PEP 735)

Use dependency groups instead of `[project.optional-dependencies]` for dev/test/lint deps that should never ship to consumers:

```toml
# Good: dev/test/docs as groups, not optional-dependencies
[dependency-groups]
dev = ["pytest>=8", "ruff", "mypy"]
docs = ["mkdocs-material"]
```

```bash
uv sync --group dev          # default; install dev group
uv sync --no-dev             # production install: skip dev group
uv sync --group docs         # docs build
```

## CI patterns

```yaml
# Good: GitHub Actions with uv
- uses: astral-sh/setup-uv@v3
  with:
    enable-cache: true
- run: uv sync --frozen
- run: uv run pytest
- run: uv run ruff check .
- run: uv run mypy src/
```

`--frozen` makes CI fail loudly when `uv.lock` was not committed alongside `pyproject.toml` changes — preventing drift between local and CI installs.

## Running scripts with inline deps (PEP 723)

For one-off scripts, declare deps inline so the script is reproducible without a project:

```python
# Good: PEP 723 inline metadata
# /// script
# requires-python = ">=3.12"
# dependencies = ["httpx", "rich"]
# ///
import httpx
from rich import print

print(httpx.get("https://example.com").status_code)
```

```bash
uv run my_script.py   # uv resolves and caches deps in an ephemeral env
```

## What to remove

When adopting uv, delete:

- `requirements.txt` / `requirements-dev.txt` (replaced by `pyproject.toml` + `uv.lock`)
- `Pipfile` / `Pipfile.lock` (pipenv)
- `poetry.lock` and `[tool.poetry]` section (Poetry — uv reads PEP 621 `[project]`)
- `setup.py` / `setup.cfg` if metadata-only (move to `[project]`)
- `.python-version` files generated by pyenv (use `uv python pin`)
- Bespoke `Makefile` venv targets — `uv sync` handles it

Keep `pyproject.toml` as the single source of truth. See `packaging/pyproject-toml-pep621.md`.

---

## Common pitfalls

❌ Don't run `pip install` inside a uv-managed project — it bypasses the lockfile and creates drift.

❌ Don't commit `.venv/`. `uv venv` recreates it deterministically from `uv.lock`.

❌ Don't mix `uv add` with manual edits to `pyproject.toml` `dependencies` array unless you also run `uv lock` after.

✅ Use `uv pip install` (the pip-compatible subcommand) only when integrating with legacy tooling that demands a pip-like interface; prefer `uv add`/`uv sync` otherwise.
