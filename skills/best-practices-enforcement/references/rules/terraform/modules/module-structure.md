# Module Structure

A Terraform "module" is any directory containing `.tf` files. The conventions below distinguish three roles — **root modules** (apply targets), **shared modules** (published, versioned, multi-consumer), and **child modules** (private subdirectories of one root) — and prescribe what belongs where.

Source: [GCP Terraform — Reusable Modules](https://docs.cloud.google.com/docs/terraform/best-practices/reusable-modules), [HashiCorp Style Guide — modules](https://developer.hashicorp.com/terraform/language/style#module-structure).

---

## One Shared Module Per Repository

A **shared module** (consumed by multiple root modules, across teams or projects) lives in its own git repository, versioned independently.

```text
# ✅ Good: one module, one repo
terraform-google-gke-cluster/
├── README.md            # generated by terraform-docs
├── main.tf
├── variables.tf
├── outputs.tf
├── versions.tf
├── examples/
│   ├── minimal/
│   └── private-cluster/
└── test/
    └── cluster_test.go  # Terratest
```

Repository naming: **`terraform-<provider>-<name>`** (`terraform-google-gke-cluster`, `terraform-aws-eks`, `terraform-kubernetes-namespace`). This matches the public registry convention so consumers can `source = "git::..."` predictably.

Don't bundle multiple unrelated modules into one repo "for convenience" — release coupling forces every consumer to bump on changes they don't care about.

---

## Child Modules (Private Subdirectories)

A **child module** is a subdirectory of a single root, never consumed externally. Group them under `modules/`:

```text
# ✅ Good: child modules live under modules/, environments under environments/
my-product-infra/
├── modules/
│   ├── network/
│   ├── iam/
│   └── workload/
└── environments/
    ├── dev/
    ├── stg/
    └── prd/
        └── main.tf       # module "network" { source = "../../modules/network" }
```

Child modules are versioned with the root repo (no SemVer tags). They exist to **factor repetition** across environments, not to publish reusable components.

---

## Shared Modules MUST NOT Configure Providers or Backends

This is the most-broken rule in the ecosystem. A shared module that declares a `provider {}` block or `backend {}` config will break the moment a consumer needs to authenticate differently, target a different project, or use a different state backend.

```hcl
# ❌ Bad: shared module hardcodes a provider — consumer can't override
# (in terraform-google-gke-cluster/main.tf)
provider "google" {
  project = "hardcoded-project"
  region  = "us-central1"
}

resource "google_container_cluster" "this" {
  # ...
}
```

```hcl
# ✅ Good: shared module declares it REQUIRES a provider, but doesn't configure one
# (in terraform-google-gke-cluster/versions.tf)
terraform {
  required_version = ">= 1.5"
  required_providers {
    google = {
      source  = "hashicorp/google"
      version = ">= 5.0"
    }
  }
}

# (in terraform-google-gke-cluster/main.tf — no provider block)
resource "google_container_cluster" "this" {
  # ...
}
```

The consuming **root module** is the only place that configures providers and backends:

```hcl
# ✅ Good: root module configures the provider and passes it to the shared module
provider "google" {
  project = var.project_id
  region  = var.region
}

module "gke" {
  source  = "git::https://github.com/example/terraform-google-gke-cluster.git?ref=v1.4.0"
  # ...inputs
}
```

### Passing aliased providers explicitly

When a root module needs multiple provider configurations (e.g. two GCP projects), pass them by alias:

```hcl
# ✅ Good: explicit provider passthrough for cross-project work
provider "google" {
  alias   = "host"
  project = var.host_project
}

provider "google" {
  alias   = "service"
  project = var.service_project
}

module "shared_vpc" {
  source = "./modules/shared-vpc"
  providers = {
    google.host    = google.host
    google.service = google.service
  }
}
```

The child module declares the aliases it needs in its own `versions.tf`; never let it default-resolve.

## Required Files in a Shared Module

| File | Purpose | Required? |
|---|---|---|
| `main.tf` | Primary resources | Yes |
| `variables.tf` | Inputs (with `type`, `description`, `validation`) | Yes |
| `outputs.tf` | Outputs (with `description`) | Yes |
| `versions.tf` | `required_version` + `required_providers` (loose floors, see `language/version-constraints.md`) | Yes |
| `README.md` | Generated by terraform-docs (see `security/pre-commit-hooks.md`) | Yes |
| `examples/<name>/` | Runnable usage examples (CI-tested) | Yes |
| `LICENSE` | Open-source license file for public modules | Yes (public) |
| `CHANGELOG.md` | Per-release notes (see `modules/semantic-versioning.md`) | Strongly recommended |

`providers.tf` and a `terraform { backend {} }` block are **explicitly forbidden** in shared modules.

---

## Module Composition: Shallow Over Deep

Prefer composing 2-3 shallow modules in a root over a single deeply nested module tree. Deep nesting (root → A → B → C) makes plan output unreadable, multiplies the surface area of `providers = {}` passthrough, and complicates `moved {}` migrations.

A good rule of thumb: a root module's `terraform graph` should fit on one screen. If it doesn't, flatten.

---

## Anti-patterns

```text
# ❌ Bad: kitchen-sink repo with 12 unrelated modules
terraform-modules/
├── gke-cluster/
├── cloud-sql/
├── pubsub/
├── slack-webhook/
└── ... 8 more
```

Each consumer gets every CHANGELOG entry, every release. Split into per-module repos.

```hcl
# ❌ Bad: child module reaches into the root's state
# (in modules/foo/main.tf)
data "terraform_remote_state" "root" {
  backend = "gcs"
  config  = { bucket = "..."; prefix = "..." }
}
```

Pass values as inputs. Remote state should be consumed only at root-module level for cross-stack data flow.
